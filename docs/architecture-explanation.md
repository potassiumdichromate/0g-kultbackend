# Architecture Explanation

Written for an engineer joining this project cold. If you only read one doc, read [../architecture/00-platform-vision.md](../architecture/00-platform-vision.md) first, then this one, then [../architecture/01-system-overview.md](../architecture/01-system-overview.md).

## The one rule that shapes everything else

**This is a platform that games plug into, not a collection of services that watch games from the sidelines.** Games own gameplay, rendering, input, and local logic. The platform owns identity, profiles, the save pipeline, security, cross-game progression, achievements, rewards, analytics, and notifications — see `00-platform-vision.md` for why (short version: ZeroDash and Warzone Warriors each independently rebuilt nearly the same auth/0G-Storage/anti-cheat code, and ZeroDash hardcoded a direct call into Warzone's API just to grant one reward — that's what happens by default without a platform to plug into).

**A second, equally important rule: neither existing game backend's source code is ever modified.** Those two constraints together are why the platform currently *observes* ZeroDash/Warzone through a polling adapter instead of owning their save pipeline outright — that's a temporary bridge dictated by constraint #2, not the target shape dictated by constraint #1. The two games are expected to migrate fully onto the platform's save pipeline (`save-service`); see `../architecture/08-migration-roadmap.md`. Don't read the current adapter-based integration as "the architecture" — read it as "the bridge until the architecture applies to these two games as well."

That distinction is why almost every other decision below looks the way it does.

## Why a Game Adapter exists at all, and why it's not the target

A naive "unified platform" would have profile-service call `GET /player/profile` on each game backend whenever it needs data, the same way ZeroDash's `crossGameService.js` already calls Warzone's API directly today. That's the anti-pattern this whole project is meant to remove: it means every game backend has to know the platform's URL, every platform service has to know every game's URL, and adding game #5 means changing code in games #1-4.

**Game Adapters** avoid that anti-pattern today — a game backend doesn't know `sync-service` exists, it just keeps serving the same public endpoints it already serves — but they're still a workaround, not the platform owning the capability. (Round 3 collapsed what were originally two near-identical per-game adapter services into one `sync-service` that reads which games to poll from Postgres — itself an instance of the same "data, not code" fix described throughout this doc.) The real target (per the vision doc) is that the platform owns the save pipeline directly via `save-service`, and no polling is needed at all because the platform receives the save in real time instead of polling for evidence one happened. Once ZeroDash/Warzone migrate, `sync-service` simply stops polling them — confirmed live: flipping a `Game.integrationMode` row in Postgres makes `sync-service` drop that game within one refresh cycle, no redeploy.

## Why NATS events instead of a shared database table

Could `reward-service` just query a `saves` table that `sync-service` writes to, instead of publishing an event? Yes, but then every consumer (`achievement-service`, `leaderboard-service`, `analytics-service`, `notification-service`, and whatever service exists a year from now) has to poll that table, and adding a consumer means understanding the producer's write pattern. A pub/sub event means the producer doesn't know or care how many consumers exist, and a new consumer is purely additive — subscribe to a subject, done. JetStream (not core NATS) closes the one real gap pub/sub has versus a database: durability. See [../architecture/05-nats-topics.md](../architecture/05-nats-topics.md) for why that distinction mattered enough to choose it deliberately.

## Why one Postgres database, when "microservices" usually means one database per service

Pure microservice doctrine says yes, split it. But this platform's entire value proposition is cross-game aggregation — a leaderboard service that can't see achievement data, or an achievement service that can't see leaderboard data, makes "shared XP" and "cross-game achievements" much harder to build for no operational benefit at this scale (two games, soon a handful, eventually 100 — but 100 games sharing a `users` table is still one logical concern, not 100). Splitting now would mean distributed transactions or eventual-consistency hacks to answer questions like "show me this user's full cross-game profile," which today is one Prisma query with a couple of `include`s. The schema documents which service owns writes to which table specifically so the split is mechanical later, not a redesign — see [../architecture/08-migration-roadmap.md](../architecture/08-migration-roadmap.md) Phase 4.

## Why Express, not the NestJS originally proposed

The original plan called for NestJS across all 8+ services, reasoning that its DI/module conventions avoid reinventing the same NATS-listener/Redis-client/JWT-guard wiring eight times. In practice, for skeletons whose entire job is to prove that wiring works end to end, NestJS's decorator/metadata machinery (and an extra layer of framework-specific debugging if something doesn't compile) added more risk than the DI convenience was worth at this stage, and it diverges from the plain-Express style both existing repos already use. Each service ended up the same shape Nest would have given it anyway — one `main.ts`, one router file, one consumer file — just without the framework. If the team grows past a handful of engineers per service, revisiting this with Nest (or staying with Express but adding a thin internal convention layer) is a reasonable next step; it's noted here so the deviation from the original plan is visible, not silent.

## Why JWT claims are deliberately identical to what already exists

`identity-service` issues `{ walletAddress, sub }` HS256 tokens with a 7-day expiry — the exact shape `authController.js` and `middleware/auth.js` already produce/consume in both repos. This wasn't the "more correct" choice (RS256 + a dedicated `aud`/`iss` claim set would be more conventional for a multi-service identity provider) — it was the *lowest-risk* choice, because it means a game's existing auth middleware accepts a platform-issued token the moment its `BROWSER_JWT_SECRET` env var is pointed at the platform's secret, with zero code changes. The more conventional approach is Phase 4 in the migration roadmap, once the lower-risk version has proven itself. This also turned out to matter for Round 2: both Unity clients' `BackendSyncManager`/`BackendService` already send `Authorization: Bearer <jwt>` in exactly this shape, confirmed by reading the actual Unity source — `save-service` accepts those tokens with zero rework.

## Round 2: why 0G Storage had to become the literal, not just nominal, source of truth

The first draft of this round's plan included a `PlayerStatistic(key, value Json)` Postgres table to generalize ZeroDash's `coins`/`characters` and Warzone's `PlayerResources`/`PlayerRambos`/etc. across games. The user corrected this directly: 0G Storage must be the *only* place actual save content lives, always binary-encoded — Postgres holds a pointer and nothing else. The `PlayerStatistic` table was dropped before being built, not refactored after — see `architecture/03-database-diagram.md` for the full reasoning. The save flow itself was also corrected mid-design: Unity → Redis (fast working copy) → encode → 0G Storage (the actual save), not Unity → encode → 0G → cache-in-Redis-afterward. The distinction matters because it's what makes "flush Redis, the save survives" true in practice, not just in a diagram — and that property was actually tested, not assumed (delete the Redis key, `GET /save` still returns the exact original JSON, recovered from the storage driver).

## Round 2 → Round 4: why per-game schemas exist, and why they no longer live in `save-service`

Per the user's explicit request, the actual Unity client source for both games was read (`Metal Black OPS/Assets` for Warzone, `TempleEscape/Assets` for ZeroDash — read-only, not modified) before designing the save payload contract. That surfaced the exact field names Unity already produces (`PlayerResources.coin`, `characters.unlocked`, etc.) and a real production consideration that wouldn't have been visible from the backend code alone: Warzone's client fires a full-profile save on 17+ different micro-events plus a 25-second autosave loop, while ZeroDash only saves once per session on game-over. Per-game schema validation is a meaningful security improvement rather than guesswork *because* it's built from that real shape — a tampered or malformed payload is rejected by structure before it's ever encoded or charged a 0G Storage write.

Round 2 put `ZeroDashSaveDataSchema`/`WarzoneSaveDataSchema` in `shared/dto`, looked up by `save-service` from a static map keyed on `gameKey`. Round 4 moved them again, this time out of shared code entirely and into each game's own `services/games/<game>-service` — see the "per-game service" section below for why. `save-service` no longer knows any game's shape at all; it trusts that whatever called it already validated.

## Round 4: why a per-game service sits in front of `save-service`, and what it is/isn't allowed to own

A naive reading of "the platform owns the save pipeline" would have Unity call `save-service` directly with a `gameKey` in the URL — which is exactly what Round 2/3 built. The user's correction: Unity should talk to *that game's own service* (`warzone-service`, `zerodash-service`), which owns two things that are genuinely that game's concern — its save shape, and its gameplay-event vocabulary (Warzone's `mission-completed{missionId, kills, timeSeconds}`) — and delegates everything else.

This is a meaningful distinction, not a renaming exercise: it's what let the save schema move out of shared code (above), and it's where a game's *specific* business logic legitimately lives going forward (e.g. a future "Survival Game" service might have its own crafting-event endpoint) without that logic leaking into `save-service`, `achievement-service`, or any other shared service. The boundary to hold the line on: a per-game service may validate, translate, and publish — it must never itself talk to 0G Storage, Postgres, or implement anti-cheat from scratch. The moment it does, that's the ZeroDash/Warzone duplication happening a third time, just one layer further down the stack.

## Round 4: why anti-cheat got a second, synchronous mode instead of staying purely async

The user pushed back on "anti-cheat only ever runs after the save already succeeded" for a specific class of event — mission completion, ranked results, tournament/NFT rewards, leaderboard submissions — where flagging fraud after the fact isn't good enough; the reward needs to not be granted in the first place. `save-service` (via an `important: true` flag the calling per-game service sets) and `warzone-service`'s `/mission-completed` endpoint both gained a synchronous gate that can reject the request with a `422` before anything is committed. It reuses the exact same ported 0G Compute client as the async path (`shared/zg-client`'s `runAntiCheat`) — this is a *mode*, not a second implementation. `verification-service`'s async pass checks whether a save was already synchronously gated and skips it, so an important save never gets charged two separate 0G Compute calls for the same decision.

## Round 4: a real bug, found by actually running this, not found by reading the code

`profile-service`'s `GAME_SAVED` consumer and `verification-service`'s `SAVE_COMPLETED` consumer both write to `UserGameProgress.metadata` — a JSON column. The first version of `profile-service`'s consumer did a blind replace of that column on every save. Live-verifying the full per-game-service chain (not just unit-level reasoning) surfaced this immediately: `verification-service`'s verdict (`computeStatus: "skipped"`) was getting silently overwritten back to `"pending"` because `profile-service`'s write happened to land after it. Fixed by merging instead of replacing. This is exactly the kind of bug that "looks fine" reading either consumer in isolation and only shows up when both run against the same row — worth remembering as a pattern (merge, don't replace, on any shared JSON column) rather than a one-off fix.

## Round 2: why anti-cheat became a shared `verification-service` instead of two copies — and why it stays separate from the old repos' anti-cheat permanently, not just for now

`ZeroGCompute.js` (anti-cheat half) is duplicated, nearly verbatim, in both `zerodash-0g-backend` and `warzone-backend-0g`. The *managed* save pipeline has no other backend to do anti-cheat for it, so that logic had to live somewhere — porting it once into `shared/zg-client` + `verification-service`, instead of writing a third copy, is the direct continuation of this project's whole premise (don't duplicate infrastructure code per game). The anti-cheat trigger threshold is also no longer hardcoded — it reads from the new `GameMetadata` table per game, a concrete instance of the "config not code" extension point.

This was explicitly confirmed, not assumed: `verification-service` and each existing repo's `ZeroGCompute.js` stay permanently separate, by design — even after ZeroDash/Warzone migrate their save pipeline onto `save-service`. There's no plan to unify them. The reason this isn't a contradiction of "the platform owns anti-cheat": once a game migrates, Unity stops calling its old backend for saves at all, so that backend's `ZeroGCompute.js` simply has nothing left to trigger on — `verification-service` becomes "the" anti-cheat for that game by replacement, not by merging logic with the old implementation.

## What "production-grade" means here, given the scope

This is a skeleton: each service proves its piece of the wiring (NATS pub/sub, Prisma access, Redis caching, JWT issuance/verification, HTTP proxying) actually runs, not a fully hardened production system. Specifically not yet done, and called out rather than silently skipped: durable JetStream pull-consumers with manual ack (current consumers are ephemeral subscriptions backed by a persistent stream — better than core NATS, not yet as robust as a tracked consumer), Achievement criteria evaluation beyond one seeded rule, real push/email delivery in `notification-service`, RS256/SSO, a real fix for `BattlePassProgress`'s platform-wide-row race once `reward-service` is horizontally scaled (see `architecture/09-security-model.md`), and a Unity-side debounce for high-frequency save clients (Warzone's, specifically). Each is named explicitly in [../architecture/08-migration-roadmap.md](../architecture/08-migration-roadmap.md) so "skeleton" doesn't quietly become "finished." What *is* done and actually verified against running infrastructure, including a real production backend's real data, is logged turn-by-turn in `../Knowledge_Base.md` — that file is the record of what was tested, not just built.
