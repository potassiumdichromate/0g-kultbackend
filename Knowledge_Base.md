## Session: 2026-06-28 (Round 5) — Per-game services as Unity's front door; synchronous TEE gate for important events

### Request

After Round 4 shipped, the user clarified the architecture further with a long, detailed message describing an AI-Arena-style model: API Gateway, Identity, Profile, **one dedicated service per game**, Leaderboard, Achievement, Reward, Analytics — explicitly distinct from what existed (Unity calling the generic `save-service` directly with just a `gameKey`). Concrete worked example given: "Warzone Service asks Profile Service: give me this player's Warzone save... Warzone never directly talks to 0G or PostgreSQL," and "Warzone publishes MISSION_COMPLETED... Achievement/Reward/Leaderboard/Analytics/Profile Service react independently... Warzone doesn't know about any of these services." Also asked for stronger anti-cheat: for "important" events specifically (mission completion, ranked results, tournament/NFT rewards, leaderboard submissions), 0G Compute TEE verification should run *before* committing, not just async afterward. Explicitly reconfirmed: build entirely inside `0g-kultbrowser`, never touch the existing repos — that was always about not touching the *existing, running* repos, not a restriction on building new services. After a clarifying summary and an explicit "yes," this was implemented in full this round.

### What was built and changed (not just designed)

- **`services/games/warzone-service` and `services/games/zerodash-service`** (new) — the reference implementation of the per-game-service tier. Each owns: that game's save-shape Zod schema (moved here, see below), validation at the door, and delegates the actual encode/upload/cache mechanic to `save-service` over an internal HTTP call. `warzone-service` additionally exposes `POST /mission-completed`, the first real (not synthetic) producer of `MISSION_COMPLETED` with actual gameplay fields (`kills`, `timeSeconds`).
- **Per-game save schemas moved out of `shared/dto` entirely**, into each new service's own `src/save-schema.ts`. `save-service` itself became fully schema-agnostic — it no longer has a `SAVE_DATA_SCHEMAS` lookup at all; its `POST/GET /save/:gameKey` contract changed to `{ data, coinSnapshot?, important? }`, trusting that whatever called it already validated. This directly resolved a Round 3 vision-doc inventory item ("self-service schema registration") that had been sitting as the one remaining open item from that round.
- **Synchronous TEE gate, a real second mode of the same anti-cheat client, not a second implementation**: `save-service` accepts `important: true` and, when set, calls `@platform/zg-client`'s `runAntiCheat` *before* encoding/uploading, rejecting with `422` on a `SUSPICIOUS` verdict. `warzone-service`'s `/mission-completed` always runs this gate (mission completion is inherently the "important" case). Both gracefully proceed (`verdict: "SKIPPED"`) with no `ZG_COMPUTE_API_KEY` configured — same pattern as the existing async path, verified live, not just asserted.
- **`verification-service`'s async consumer now skips saves already synchronously gated** (checks `payload.computeStatus !== "pending"`) — otherwise an important save would trigger two separate 0G Compute calls for one decision.
- **API Gateway**: added `/api/v1/play/<gameKey>` routing to the new per-game services, kept distinct from the existing `/api/v1/games/<gameKey>` legacy passthrough (same `gameKey` can exist in both during the migration window).
- **Root `package.json` workspaces glob** needed `"services/games/*"` added — `services/*` doesn't reach two directories deep; npm silently wouldn't have picked up the new services without this.
- **A real concurrency bug, found by actually running the full chain, not by reading the code**: `profile-service`'s `GAME_SAVED` consumer did a blind metadata *replace* on `UserGameProgress`, racing with `verification-service`'s update. First live run of the full per-game-service chain showed `computeStatus` reverting from `"skipped"` back to `"pending"` after `verification-service` had already logged success — `profile-service`'s write landed after it and clobbered it. Fixed by merging metadata instead of replacing, with `computeStatus` treated as "verification-service's field, only seed a default if nothing has set it yet." Reran the full test after the fix; confirmed `computeStatus`/`verdict` survive correctly.

### Verified live, end to end, in one chained test (not five separate claims)

A single throwaway wallet, through `identity-service` login, then: (1) `POST /save` on `warzone-service` with a real Warzone-shaped payload → `201`, round-tripped byte-for-byte on `GET /save`; (2) a deliberately malformed save (`{PlayerResources: {coin: -999}}`) → rejected `400` by `warzone-service`'s own schema, confirmed it never reached `save-service` at all; (3) `POST /mission-completed {missionId, kills: 48, timeSeconds: 480}` → `201`, `verdict: "SKIPPED"` (no compute key configured); (4) `GET /profile/:wallet` on `profile-service` showed `achievementCount: 2` (`first_save` from the save, `warzone_first_blood` — a new achievement this service seeds itself, scoped to `gameId: warzone` — from the mission) and `xpTotal: 100` (50 from each); (5) direct Postgres query confirmed `UserGameProgress.metadata` correctly retained `computeStatus: "skipped"` / `verdict: "SKIPPED"` after the profile-service fix, where it had been incorrectly reset to `"pending"` before the fix.

### Open items added this round

- `zerodash-service` has no gameplay-event endpoint yet (ZeroDash has no granular mission/level concept in its current save model) — fine for now, noted as the asymmetry between the two reference implementations, not a gap to fix speculatively.
- The per-game save schema is still hand-written Zod code inside each service (not a literal "submit a JSON schema, no code" admin flow) — the vision doc's inventory entry for this was updated to say the *shared-code coupling* is resolved, while flagging this narrower remaining gap explicitly rather than overclaiming full self-service.

## Session: 2026-06-28 (Round 2) — Managed save pipeline, verification service, schema redesign

Same conversation as Round 1, continued. User supplied three real Mongoose schema files (`Zerodash_Model.js`, `IAPPurchase_forWarzone.js`, `PlayerProfileModel_ForWarzone.js`) and asked for the architecture to be *updated, not recreated*, with: (1) Unity sending/receiving plain JSON only — backend owns encode/compress/0G upload/download; (2) the existing (but not-yet-built-here) 0G Compute TEE anti-cheat logic formalized as a platform "Verification Service"; (3) a redesigned Postgres schema using the real schemas, normalized, no per-game columns; (4) expanded NATS events, a security-model doc, an incremental migration plan, and full doc updates. Mid-planning, the user corrected the design twice (both addressed before any code was written — see "Corrections" below) and then asked for the *actual Unity client source* (`Metal Black OPS/Assets` for Warzone, `TempleEscape/Assets` for ZeroDash) to be read before finalizing the save contract.

### Corrections received during plan-mode review (both resolved before implementation)

1. **0G Storage must be the only source of truth for actual save content, always binary-encoded.** My first plan draft included a `PlayerStatistic(key, value Json)` Postgres table generalizing both games' profile fields — the user rejected this directly ("you get that?"). Dropped before being built. Postgres in Round 2 holds only pointers (`UserGameProgress.rootHash`) and platform-computed records, never save content — restated as a "ground rule" comment block at the top of `shared/db/prisma/schema.prisma` and in `architecture/03-database-diagram.md`.
2. **Save flow ordering**: Unity → JSON → Redis (fast working copy, written first) → encode → 0G Storage (the actual save, not the cache). My first draft had the encode/upload happen before mentioning Redis explicitly enough; corrected in the plan before `save-service` was written, not after.
3. **Read the real Unity clients before finalizing the save JSON contract** — done via two read-only Explore-agent passes over `Metal Black OPS/Assets/_Assets/Web3Integ/{ZGSaveManager.cs, BackendSyncManager.cs}` (Warzone) and `TempleEscape/Assets/_TempleEscape/Scripts/Savestate/{ZGSaveManager.cs, BackendService.cs, GameBootstrapper.cs}` (ZeroDash). Findings directly shaped the implementation — see `docs/repository-mapping.md`'s "Unity client mapping" section for the full detail. Highlights: Warzone's client saves on 17+ micro-events plus a 25s autosave (full-profile re-upload each time); ZeroDash's client saves once per session on game-over; both already send `Authorization: Bearer <jwt>` in the exact shape `identity-service` issues.

### What was built

- **Postgres schema additions**: `GameMetadata` (per-game static config — `reward-service`'s and `verification-service`'s thresholds now read from here instead of hardcoded constants), `BattlePassProgress` (platform-wide, `gameId: null`, cross-game XP), `IapPurchase` (normalized from the supplied `IAPPurchase_forWarzone.js`, schema-only, no producer yet), `SecurityAuditLog` (reserved for synchronous identity-service writes — schema added, the actual write call sites are a documented open item, not yet wired). Migration `20260628103427_round2_save_pipeline` applied to the live local Postgres; verified all 15 tables exist via `psql \dt`.
- **`shared/zg-client`** (new shared package): `storage.ts` ports `ZeroGStorage.js`'s upload/download pattern (temp file, Merkle root, indexer rotation, retry) behind a `StorageDriver` interface with two implementations — the real 0G driver (lazy-loads `@0gfoundation/0g-storage-ts-sdk` + `ethers`, selected when `ZG_PRIVATE_KEY`/`ZG_RPC_URL` are set) and a local-disk driver (sha256-keyed file storage, selected otherwise). `compute.ts` ports `ZeroGCompute.js`'s anti-cheat client (OpenAI-compatible call, rootHash-echo binding check, graceful skip with no API key).
- **`shared/dto/src/save-data.dto.ts`**: `ZeroDashSaveDataSchema`/`WarzoneSaveDataSchema`, built field-for-field from the real Unity `PlayerSaveData`/save-JSON shapes found above, plus `extractCoinSnapshot` per game.
- **`shared/events`**: added `PLAYER_LOGIN`, `SAVE_REQUESTED`, `SAVE_COMPLETED` (a superset of `GameSavedPayloadSchema`, carrying `previousCoinSnapshot`/`previousSaveIndex` so Verification Service can compute a real delta), `SAVE_VALIDATED`, `GAME_INSTALLED`, `PROFILE_UPDATED`, `PLAYER_ONLINE`/`OFFLINE`.
- **`services/save-service`** (new): `POST /save/:gameKey` / `GET /save/:gameKey`. Wallet always from verified JWT, never the body. `saveIndex` always server-computed (no client-supplied anti-rollback header to spoof, unlike the legacy binary endpoints). Per-game Zod validation before anything is encoded. - **`architecture/09-security-model.md`** (new): wallet-from-JWT-never-body, server-computed saveIndex, per-game schema validation, sync-structural-vs-async-semantic validation split, rootHash ownership, synchronous security audit logging, and the documented (not fixed) `BattlePassProgress` null-uniqueness limitation under horizontal scaling.

### Analysis findings (full detail in the two Explore-agent reports from this session; summarized here)

- Both repos: Node.js/Express + MongoDB(Mongoose), `ethers` v6, identical SIWE-nonce + HS256-JWT auth flow, identical `@0gfoundation/0g-storage-ts-sdk` usage, identical `PlayerSaveAnchor.sol` contract + `disperser.proto` gRPC client, near-identical `PlayerSaveRecord` schema (rootHash, saveIndex, daStatus, computeStatus, checksum), near-identical trust-score/badge UX endpoints.

This is the concrete anti-pattern the new platform's event-driven design replaces.
- Notable anti-patterns flagged across both repos (see full reports for file:line detail): `console.log`-only logging, hardcoded CORS allowlists, secrets with insecure dev defaults that only warn instead of failing, no audit logging, business logic embedded in controllers, no checksum re-verification on download.

### Architectural decisions (reasoning recorded in `docs/architecture-explanation.md` and `architecture/08-migration-roadmap.md` — not duplicated here)

1. **Zero-touch Game Adapters** poll each game's existing public `/player/leaderboard/decentralized` + `/player/save/metadata` endpoints in Phase 1 — no code change to either existing repo. A native-SDK push model is Phase 2, opt-in per game.
2. **Identity Service** issues JWTs with the exact claim shape (`{walletAddress, sub}`, HS256, 7-day) both existing repos' auth middleware already expects — drop-in, not a forced migration. RS256/true-SSO is Phase 3.
3. **Single shared Postgres + Prisma schema** for v1 (not DB-per-service) — deliberate tradeoff given the platform's core value is cross-game aggregation; documented split path for later (Phase 4) in the migration roadmap.
4. **NATS JetStream**, not core pub/sub — durability matters once rewards/achievements carry real value.
5. **Express + TypeScript instead of the originally-planned NestJS** for service skeletons — a deviation made during implementation (see "Deviations from the approved plan" below), favoring a smaller dependency surface for a skeleton whose job is to prove wiring works.

The full design was written to a plan file and approved via the plan-mode workflow before any code was written (`EnterPlanMode` → plan reviewed → `ExitPlanMode` approved).
