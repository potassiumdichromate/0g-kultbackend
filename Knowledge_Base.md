### What was built

- **Postgres schema additions**: `GameMetadata` (per-game static config — `reward-service`'s and `verification-service`'s thresholds now read from here instead of hardcoded constants), `BattlePassProgress` (platform-wide, `gameId: null`, cross-game XP), `IapPurchase` (normalized from the supplied `IAPPurchase_forWarzone.js`, schema-only, no producer yet), `SecurityAuditLog` (reserved for synchronous identity-service writes — schema added, the actual write call sites are a documented open item, not yet wired). - **`shared/dto/src/save-data.dto.ts`**: `ZeroDashSaveDataSchema`/`WarzoneSaveDataSchema`, built field-for-field from the real Unity `PlayerSaveData`/save-JSON shapes found above, plus `extractCoinSnapshot` per game.
- **`shared/events`**: added `PLAYER_LOGIN`, `SAVE_REQUESTED`, `SAVE_COMPLETED` (a superset of `GameSavedPayloadSchema`, carrying `previousCoinSnapshot`/`previousSaveIndex` so Verification Service can compute a real delta), `SAVE_VALIDATED`, `GAME_INSTALLED`, `PROFILE_UPDATED`, `PLAYER_ONLINE`/`OFFLINE`.
- **`services/save-service`** (new): `POST /save/:gameKey` / `GET /save/:gameKey`. Wallet always from verified JWT, never the body. `saveIndex` always server-computed (no client-supplied anti-rollback header to spoof, unlike the legacy binary endpoints). Per-game Zod validation before anything is encoded. Redis working copy written first; msgpack-encode + gzip-compress; upload via the storage driver; Postgres gets only the pointer + a couple of denormalized scalars; publishes `SAVE_REQUESTED`/`SAVE_COMPLETED`/`GAME_SAVED` (the last two as the same payload, so every Round 1 consumer picks up managed saves with zero changes).
- **`services/verification-service`** (new): consumes `SAVE_COMPLETED`, re-applies the `saveIndexDelta >= 1 || coinDelta > threshold` heuristic from `ZeroGCompute.js` with the threshold read from `GameMetadata`, calls the ported compute client, merges `computeStatus`/`verdict` into the existing `UserGameProgress.metadata` (without clobbering other fields), publishes `SAVE_VALIDATED`.
- **XP/Battle Pass wiring**: `achievement-service` now publishes `platform.user.xp_gained` (fixed 50 XP) on every achievement unlock — this event existed as a schema since Round 1 with no producer. `profile-service` added an `xp.consumer.ts` that's the sole writer of `User.xpTotal`/`level` (simple `level = floor(xpTotal/100)+1` formula) and publishes `platform.profile.updated`. `reward-service` added a `battlepass.consumer.ts` that independently consumes the same `xp_gained` event to advance a platform-wide `BattlePassProgress` row — completing the chain: save → achievement → XP → battle pass → reward → profile → analytics → notification.
- **`architecture/09-security-model.md`** (new): wallet-from-JWT-never-body, server-computed saveIndex, per-game schema validation, sync-structural-vs-async-semantic validation split, rootHash ownership, synchronous security audit logging, and the documented (not fixed) `BattlePassProgress` null-uniqueness limitation under horizontal scaling.
- **Migration roadmap renumbered**: Phase 3 is now "Managed save pipeline" (this round); RS256/SSO moved to Phase 4, DB split to Phase 5.

## Session: 2026-06-28 — Initial platform architecture + scaffold

### Request

User (Senior/Principal Engineer persona) asked for a complete architecture analysis of two existing, production game backends —
`C:\Users\RENTKAR\Desktop\0g-ai\0g-Zerodash\zerodash-0g-backend` (ZeroDash) and
`C:\Users\RENTKAR\Desktop\0g-ai\0g-Warzone\warzone-backend-0g` (Warzone Warriors) —
and a new, unified-platform architecture for an upcoming multi-game "Kult Browser" (Steam/Xbox-style launcher for Unity WebGL games). Hard constraint: neither existing repo may be modified; all new work goes only inside `0g-gamebrowser/0g-kultbrowser`.

### Analysis findings (full detail in the two Explore-agent reports from this session; summarized here)

- Both repos: Node.js/Express + MongoDB(Mongoose), `ethers` v6, identical SIWE-nonce + HS256-JWT auth flow, identical `@0gfoundation/0g-storage-ts-sdk` usage, identical `PlayerSaveAnchor.sol` contract + `disperser.proto` gRPC client, near-identical `PlayerSaveRecord` schema (rootHash, saveIndex, daStatus, computeStatus, checksum), near-identical trust-score/badge UX endpoints.
- Only genuinely game-specific: the player profile schema (ZeroDash: coins/highScore/characters; Warzone: campaign/rambos/guns/grenades/daily quests) and the 0G-Compute anti-cheat/AI system prompts.
- ZeroDash already has a hand-rolled cross-game mechanism: `crossGameService.js` hardcodes 4 other games' backend URLs, and `warzoneGunRewardClient.js` hardcodes Warzone's URL plus a literal shared-secret string committed to source (`'warzone-gun-cross-game-reward-v1'`) to grant a gun unlock. This is the concrete anti-pattern the new platform's event-driven design replaces.
- Notable anti-patterns flagged across both repos (see full reports for file:line detail): `console.log`-only logging, hardcoded CORS allowlists, secrets with insecure dev defaults that only warn instead of failing, no audit logging, business logic embedded in controllers, no checksum re-verification on download.

### Architectural decisions (reasoning recorded in `docs/architecture-explanation.md` and `architecture/08-migration-roadmap.md` — not duplicated here)

1. **Zero-touch Game Adapters** poll each game's existing public `/player/leaderboard/decentralized` + `/player/save/metadata` endpoints in Phase 1 — no code change to either existing repo. A native-SDK push model is Phase 2, opt-in per game.
2. **Identity Service** issues JWTs with the exact claim shape (`{walletAddress, sub}`, HS256, 7-day) both existing repos' auth middleware already expects — drop-in, not a forced migration. RS256/true-SSO is Phase 3.
3. **Single shared Postgres + Prisma schema** for v1 (not DB-per-service) — deliberate tradeoff given the platform's core value is cross-game aggregation; documented split path for later (Phase 4) in the migration roadmap.
4. **NATS JetStream**, not core pub/sub — durability matters once rewards/achievements carry real value.
5. **Express + TypeScript instead of the originally-planned NestJS** for service skeletons — a deviation made during implementation (see "Deviations from the approved plan" below), favoring a smaller dependency surface for a skeleton whose job is to prove wiring works.

The full design was written to a plan file and approved via the plan-mode workflow before any code was written (`EnterPlanMode` → plan reviewed → `ExitPlanMode` approved).

### Deviations from the approved plan (flagged explicitly per instructions, not silently made)

- **NestJS → Express.** The approved plan specified NestJS for DI/guard/module conventions across 8+ services. During implementation this was changed to plain Express + TypeScript to keep the dependency surface small and reduce framework-specific failure modes for what is explicitly a skeleton, and to match the existing repos' own Express conventions. Documented with reasoning in `docs/architecture-explanation.md`. If the team scales past a handful of engineers per service, revisit.
- JetStream consumers in the skeleton are ephemeral `nc.subscribe()` calls against durable streams, not yet promoted to true durable pull-consumers with manual ack — called out as the first hardening step in `architecture/05-nats-topics.md` and `architecture/08-migration-roadmap.md`, not hidden.
